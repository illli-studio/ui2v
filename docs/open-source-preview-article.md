# ui2v 开源预告：把 UI、动画和代码真正渲染成视频

如果你做过产品演示、数据可视化短片、动态海报、AI 生成视频，或者任何需要“让界面动起来”的内容，你大概率遇到过一个很现实的问题：

我们已经可以很快写出一个漂亮的网页界面，也可以让 AI 生成一段动画代码，甚至可以用 Canvas、Three.js、D3、GSAP、Pixi、Lottie 等各种前端技术做出非常丰富的视觉效果。可是当我们想把这些东西稳定地导出成一个视频文件时，事情往往突然变复杂。

浏览器里能跑，导出时不一定能跑。

预览时看起来正常，渲染成视频后可能掉帧、错位、时间不准。

一个模板可以手动录屏，但如果要批量生成上百条视频，或者让 AI 自动生成 JSON 再交给机器渲染，就需要一套更可靠的生产流程。

ui2v 就是为了解决这个问题而做的。

它的目标很直接：让 UI、动画代码和 JSON 描述，能够通过一条稳定、可重复、可自动化的流程，渲染成真正的视频文件。

## ui2v 是什么

ui2v 是一个开源的命令行视频渲染器。它读取一份动画 JSON 文件，启动本地 Chrome 或 Edge 浏览器，在真实浏览器环境中渲染每一帧画面，然后使用 WebCodecs 编码成 MP4 视频并写入磁盘。

可以把它理解成一台“专门为 UI 动画准备的视频渲染机”：

- 输入是一份结构化的 JSON 动画项目。
- 渲染环境是真实浏览器，而不是模拟出来的画布。
- 动画可以使用 Canvas、自定义代码和浏览器原生能力。
- 输出是可以直接播放、上传、交付的 MP4 文件。

它不是一个传统的视频剪辑软件，也不是一个只能拖拽模板的页面工具。ui2v 更像是一套底层渲染基础设施：它把“界面如何变化”“每一帧是什么时间点”“哪些依赖需要加载”“最后如何编码成视频”这些事情拆开，并用清晰的工程结构连接起来。

这让它特别适合三类场景：

第一类是 AI 生成动画。AI 可以生成 JSON、生成分段脚本、生成自定义 Canvas 代码，然后 ui2v 负责把这些内容变成真正的视频。

第二类是产品和数据内容自动化。比如批量生成产品展示视频、数据报表动画、社交媒体短片、动态广告素材、课程可视化片段。

第三类是开发者自己的可编程视频系统。你不必把所有东西塞进一个庞大的桌面应用里，而是可以通过 CLI、脚本、CI 或后端服务来调用渲染流程。

## 为什么要用浏览器来渲染视频

ui2v 的一个核心选择是：尽量让动画在浏览器里渲染。

这听起来很自然，因为今天很多动态视觉本来就是为浏览器生态写的。Canvas、DOM、WebGL、CSS、WebCodecs，以及大量前端动画库，都是在浏览器环境里最自然、最完整。

过去如果想在 Node.js 里渲染画布，常常会碰到 native canvas、FFmpeg、Electron、系统编码器、平台依赖等问题。不同系统、不同机器、不同安装环境，可能会带来一堆额外麻烦。

ui2v 的主路径尽量避开这些重依赖：

- 不需要 Electron 作为主运行时。
- 不依赖 node-canvas 作为主要渲染方案。
- 不把 FFmpeg 作为默认必需项。
- 使用 Puppeteer 启动本地浏览器。
- 在浏览器中用 WebCodecs 完成 MP4 编码。

这样做的好处是很明显的：动画在哪个环境里本来就应该运行，就让它在哪个环境里运行。渲染器不需要重新发明一个“半浏览器”，也不需要把本来属于前端生态的能力硬搬到别处。

更重要的是，预览和导出可以尽量共享同一套时间线和运行时逻辑。你在预览里看到的内容，和最后导出的视频，应该来自同一个动画计划，而不是两套各算各的系统。

## 它现在已经能做什么

当前版本的 ui2v 已经具备了一条完整的本地渲染链路。

你可以用命令行检查环境：

```bash
node packages/cli/dist/cli.js doctor
```

这个命令会确认本地浏览器是否支持 WebCodecs，以及默认的 AVC/H.264 编码配置是否可用。

你可以校验动画 JSON：

```bash
node packages/cli/dist/cli.js validate animation.json
```

你可以打开浏览器预览：

```bash
node packages/cli/dist/cli.js preview animation.json
```

你也可以直接渲染 MP4：

```bash
node packages/cli/dist/cli.js render animation.json -o output.mp4
```

如果你想看运行时在某些时间点到底算出了什么状态，还可以使用：

```bash
node packages/cli/dist/cli.js inspect-runtime animation.json --time 0 --time 1
```

这对调试 AI 生成动画尤其有用。因为很多时候问题不是“画不出来”，而是某个时间点、某个分段、某个图层的状态不符合预期。inspect-runtime 的意义，就是让动画不再只是黑盒。

项目里也已经包含了几个示例：

- basic-text：基础文字动画，用来验证最小可用路径。
- product-showcase：产品展示短片，包含背景、网格、产品主体、信息卡片、按钮式视觉元素和入场动画。
- kitchen-sink：一个更长的综合压力示例，展示多种视觉风格和库能力的组合。
- runtime-core 示例：展示新的 uiv-runtime JSON 结构、分段视频、运行时命令流和多种未来适配方向。

## 可以做出哪些效果

ui2v 最适合的不是传统拍摄素材剪辑，而是“由代码和数据生成的动态画面”。它能做出来的效果可以非常宽。

比如，产品发布视频。

你可以让一个产品卡片从画面外滑入，背景网格缓慢移动，产品主体带一点轻微旋转，旁边逐步出现卖点卡片，最后亮出“导出 MP4”或“立即发布”之类的视觉按钮。整个过程不是靠手动关键帧拖出来，而是由 JSON 和代码描述。

比如，数据可视化视频。

柱状图可以随着时间跳动，折线图可以扫过画面，散点和粒子可以按节奏聚合，数据标签可以逐个出现。适合做日报、周报、金融数据、运营看板、排行榜、趋势解释视频。

比如，科技感动效。

星空背景、赛博城市、发光轨道、线框立方体、动态粒子、矩阵字符、扫描线、HUD 面板、数字孪生风格界面，这些都可以用 Canvas 或自定义代码生成。

比如，3D 和伪 3D 演示。

现在的示例里已经有分子结构、旋转线框、投影计算等 Canvas 实现。未来随着 Three.js 等适配器路线完善，真正的 3D 场景、产品模型、空间 UI、地图和地球可视化，都可以进入同一条时间线。

比如，AI 自动生成的分镜视频。

Runtime Core 支持按时间段组织视频。一个 9 秒视频可以被拆成 0 到 3 秒、3 到 6 秒、6 到 9 秒三个片段。每个片段都可以拥有自己的代码、依赖、参数和局部时间。AI 很适合生成这种结构：先生成分镜，再为每段生成对应的视觉代码，最后交给 ui2v 渲染。

比如，动态模板批量生成。

同一个模板可以替换标题、图片、颜色、数据、时长和分辨率，生成大量不同版本的视频。对内容运营、广告素材、教育短视频、SaaS 产品演示、自动化报告都很有价值。

这些效果背后的共同点是：画面不是一次性的手工录屏，而是可描述、可复现、可检查、可批量生产的。

## 一份 JSON 如何变成视频

普通人可以把 ui2v 的工作流程想象成一条流水线。

第一步，读取项目文件。

项目文件是一份 JSON，里面写着视频多长、帧率多少、分辨率多大、背景是什么、有哪些图层、每个图层从什么时候开始到什么时候结束，以及图层要怎么画。

第二步，规范化。

不同格式的项目会被整理成统一的内部结构。这个结构叫 composition 和 scene graph，也就是“合成”和“场景图”。它告诉系统：画面里有哪些节点，它们的层级关系是什么，谁先画，谁后画。

第三步，计算时间线。

视频不是一张图，而是一连串帧。比如 6 秒、30 fps 的视频，就有 180 帧。Runtime Core 会为每一帧计算准确的时间点、展示时间戳、持续时间、关键帧信息，以及这一帧属于哪个视频片段。

第四步，加载依赖并渲染。

如果某个片段需要 GSAP、Three.js、D3 或其他库，运行时可以知道这个时间段需要哪些依赖。浏览器加载需要的代码后，渲染引擎把这一帧画到 Canvas 上。

第五步，编码成 MP4。

每一帧被送进 WebCodecs 编码器，最终封装成 MP4，写到本地磁盘。

这就是 ui2v 的核心路线：

```text
JSON 项目
  -> 解析和校验
  -> Runtime Core 计算场景和时间线
  -> Puppeteer 启动浏览器
  -> 浏览器渲染 Canvas 帧
  -> WebCodecs 编码 MP4
  -> 输出视频文件
```

## 项目的内部结构

ui2v 现在采用 monorepo 结构，核心包分得比较清楚。

`@ui2v/core` 负责基础类型、JSON 解析、校验和共享工具。它像是项目入口处的质检员，先确认输入是不是合理。

`@ui2v/runtime-core` 是长期核心。它不负责画图，而是负责理解动画：场景图、时间线、帧计划、分段计划、依赖计划、适配器路由、绘制命令流。它的目标是让预览、检查和导出都基于同一套确定性的时间模型。

`@ui2v/engine` 负责浏览器里的渲染能力。它包含 Canvas 渲染、模板适配、自定义代码运行、绘制命令执行，以及 WebCodecs 导出相关能力。

`@ui2v/producer` 负责把 Node.js、Puppeteer、本地静态服务、浏览器运行时和最终文件输出串起来。它是实际生产视频的管线。

`ui2v CLI` 则是开发者直接使用的命令行入口。比如 doctor、validate、preview、render、inspect-runtime、init、info 都在这里。

这种拆分的好处是，项目不会被锁死在某一种运行方式里。今天可以是本地 CLI，明天可以是服务端队列，后天也可以有在线 playground、VS Code 插件、云渲染或更多图形后端。

## Runtime Core 为什么重要

很多视频工具刚开始看起来都能“画出东西”，但真正难的是一致性。

预览是一套逻辑，导出又是一套逻辑，就很容易出现问题。比如预览第 3 秒刚好转场，导出时却偏了一帧。比如一个动画在 30 fps 下没问题，换成 60 fps 就节奏错了。比如一个分段视频的边界，在浮点数计算里出现细微误差。

Runtime Core 要解决的就是这类问题。

它会生成权威的帧计划。每一帧的 index、renderTime、presentationTime、duration、timestampUs、durationUs、keyframe 都是明确的。

它也能理解视频分段。比如 9 秒、30 fps、三个 3 秒片段的视频，可以明确得到：

```text
opening: frames 0-89
middle:  frames 90-179
resolve: frames 180-269
```

这件事对 AI 生成视频非常关键。因为 AI 经常会按“第一段、第二段、第三段”来组织内容。如果运行时也能理解这种结构，渲染、预加载、进度显示、调试和错误定位都会更可靠。

Runtime Core 还在为未来的多渲染器做准备。一个视频里可能有 Canvas2D 图层，也可能有 Three.js 图层、Pixi 图层、Lottie 图层、DOM 图层。它们可以共享同一条时间线，但由不同适配器负责渲染。这样 ui2v 就不只是一个 Canvas 工具，而是可以逐步成长为一个多后端的视频运行时。

## 为什么它适合 AI 视频时代

现在很多人谈 AI 视频，第一反应是文生视频模型。但还有另一条非常重要的路线：AI 生成结构化视频。

也就是说，AI 不直接“想象”出每一个像素，而是生成一个可执行的动画计划：

- 这个视频有几个片段。
- 每个片段讲什么。
- 每个图层是什么。
- 元素什么时候出现。
- 使用哪些数据。
- 背景、文字、图表和动效如何变化。
- 最后导出成什么尺寸和帧率。

这种方式有几个优势。

它更可控。文字错了可以改文字，颜色错了可以改颜色，数据错了可以换数据，不需要整段视频重抽奖。

它更适合产品内容。UI、图表、代码演示、知识讲解、动态海报，本来就需要清晰、准确、可编辑。

它更容易批量化。一个模板加一组数据，可以生成很多条不同视频。

它更容易开源协作。社区可以贡献模板、适配器、示例、检查器、编辑器和插件。

ui2v 想做的，就是给这条路线一个足够扎实的渲染底座。

## 开源之后希望它变成什么

这次开源不是把一个孤立工具丢出来，而是希望把 ui2v 变成一个可以共同扩展的生态基础。

短期内，项目会继续强化稳定性：

- 更好的错误提示。
- 更可靠的浏览器依赖加载。
- 更完善的示例测试。
- 更清晰的 MP4 输出验证。
- 更干净的临时文件管理。

中期会补齐更适合生产环境的能力：

- 离线或 vendor 模式，让浏览器端依赖不依赖外部 CDN。
- 更适合长视频和高分辨率视频的流式或分块输出。
- 批量渲染。
- WebM 等更多输出格式。
- 音频混合。
- 模板库。
- 插件系统。

长期来看，ui2v 可以继续扩展到：

- 在线 playground。
- 文档站。
- 社区模板市场。
- VS Code 扩展。
- 云端或分布式渲染。
- 多适配器图形运行时。

最理想的状态是：开发者、设计师和 AI 系统都可以围绕同一种结构化视频格式协作。有人写模板，有人写渲染适配器，有人做编辑器，有人做自动生成器，有人把它接入自己的内容生产系统。

## 现在的边界

ui2v 目前还处在早期阶段，但核心链路已经跑通。

当前主要输出是 MP4。默认编码是 AVC/H.264，因为这是 Chromium WebCodecs 里最通用、兼容性最好的路线。HEVC 可以请求使用，但取决于本地浏览器是否支持。

浏览器里的部分 ESM 依赖目前通过固定 CDN 地址加载，后续需要离线/vendor 模式来提升 CI 和受限网络环境下的可复现性。

长视频或高分辨率视频当前仍然会把编码结果以 base64 形式从浏览器传回 Node，再写入磁盘。这条路径简单直接，但对内存不够友好。未来会改成流式或分块传输。

这些边界并不是回避，而是开源前明确告诉大家：项目已经能做什么，哪些地方还在路上，欢迎社区一起把它补强。

## 一句话总结

ui2v 想做的是：把 UI 动画、代码动画、数据动画和 AI 生成动画，变成一条开放、可检查、可重复、可批量生产的视频渲染流程。

它不是只为了做一个漂亮 demo，而是为了让“从 JSON 到视频”这件事真正工程化。

当一个动画能被结构化描述，它就可以被 AI 生成，被程序修改，被命令行渲染，被测试验证，被批量生产，也可以被社区持续改进。

这就是 ui2v 开源的意义。

我们希望它成为一个面向未来的 UI Video Renderer：让界面不只是能显示在屏幕上，也能稳定地走进视频、内容生产、自动化和 AI 创作流程里。
